# Interview Generator — Technical Interview Evaluation with LLMs

> An interview simulator that **generates questions with an LLM**, **evaluates answers using NLP metrics**, and **adapts question difficulty in real time** according to the user's performance.

## Overview

Most question-answering systems simply compare your response against a reference solution. **Interview Generator** breaks every answer down into four measurable dimensions, computes an overall score, and uses that score to determine **which question should come next**.

The result is an adaptive practice environment: if you consistently perform well, the interview becomes more challenging; if you struggle, it lowers the difficulty and reinforces weaker concepts.

| Component | Purpose |
|---|---|
| **Question Generation (RAG)** | Retrieves relevant context from a ChromaDB vector database and prompts the LLM to generate a question at the current difficulty level. |
| **Hybrid Evaluation** | Combines four NLP metrics into a final score between 0 and 1: semantic, numerical, conceptual, and reasoning quality. |
| **Adaptive Difficulty** | Streak-based progression logic that promotes or demotes the user based on overall performance. |
| **Feedback** | Side-by-side comparison, LLM critique, step-by-step solution (Chain-of-Thought), and theoretical explanations grounded in real literature. |

## Hybrid Evaluation

Each answer is evaluated across four dimensions and combined using fixed weights ([`evaluator.py`](src/project/metrics/evaluator.py)):

| Dimension | Weight | Method |
|---|:--:|---|
| **Numerical Validation** | **0.60** | `SymPy` + regular expressions. Verifies numerical correctness with configurable tolerance. |
| **Semantic Similarity** | 0.15 | `Sentence-BERT` embeddings (`all-mpnet-base-v2`). Accepts paraphrases and synonyms. |
| **Reasoning Structure** | 0.15 | Heuristics based on logical connectors, sequential reasoning, and technical writing quality. |
| **Concept Coverage** | 0.10 | `KeyBERT` + `spaCy`. Measures whether key concepts from the reference solution are present. |

The heavy emphasis on numerical validation is intentional. Comparing free-form text alone cannot reliably distinguish a correct answer from one that merely *sounds* convincing. If the user reaches the correct numerical solution, they should pass. The remaining metrics refine the score rather than determine it.

## Adaptive Difficulty

Difficulty changes are based on **performance streaks**, not on the outcome of a single question ([`app.py`](src/project/app.py)):

- **Increase** the difficulty after **2 consecutive answers** scoring **≥ 0.85**.
- **Immediately decrease** the difficulty after any answer scoring **< 0.45**.
- Any intermediate score keeps the current level and **resets the streak**.

This avoids abrupt oscillations caused by isolated good or bad answers.

## Architecture

The backend is built with **FastAPI** and organized into three layers: a lightweight frontend (Jinja2 + JavaScript), generation and evaluation services, and **Redis** for session state management.

The lifecycle of a question is as follows:

1. The backend retrieves the current difficulty from Redis and requests a new question from the **QuestionGenerator**, which fetches relevant context from ChromaDB and refines it with the LLM.
2. The user submits an answer. It is stored immediately while the evaluation is delegated to a FastAPI **BackgroundTask**.
3. The **Evaluator** computes the four NLP metrics asynchronously (the expensive step, typically taking 5–10 seconds) and updates Redis with the results. The interface remains responsive throughout.
4. The overall score updates the user's streak and, if necessary, adjusts the difficulty of the next question.
5. Once the interview finishes, all session metrics are consolidated into a per-question performance report.

## Getting Started

**Requirements:** Python 3.10–3.14 (Python 3.15+ is not yet supported by `torch` and `transformers`) and optionally Redis. Without Redis, the application still runs in a degraded mode using in-memory storage and a single worker.

```bash
git clone https://github.com/pabloChantada/TAPL.git
cd TAPL

python -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

Create a `.env` file based on [`.env.example`](.env.example):

```bash
LLM_PROVIDER="GEMINI"          # GEMINI | GROQ | DEEPSEEK
GEMINI_API_KEY="your_api_key"
THEORY_BOOKS="files/book_id"   # Required only for the theory module (PDF RAG)
```

Build the vector database and launch the application:

```bash
./scripts/load_db.sh           # Builds data/chroma_db_* (takes a few minutes)
./scripts/run_app.sh           # Detects Redis: 4 workers if available, otherwise 1
```

The web interface will be available at **http://localhost:8000**.

Users can choose the number of questions and the initial difficulty before starting the interview. If they get stuck, the **Hint** button provides additional context without revealing the complete solution.

## API

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/datasets` | Returns the available datasets (SQuAD, CoachQuant). |
| `POST` | `/api/interview/start` | Starts a new interview session with the selected dataset, difficulty, and number of questions. |
| `GET` | `/api/interview/question/{session_id}` | Retrieves the next question at the appropriate difficulty level. |
| `POST` | `/api/interview/answer` | Submits an answer and launches background evaluation. |
| `POST` | `/api/interview/hint` | Generates a contextual hint without revealing the solution. |
| `POST` | `/api/feedback` | Returns an LLM-generated critique of the user's answer. |
| `POST` | `/api/explanation` | Generates a step-by-step solution using Chain-of-Thought prompting. |
| `POST` | `/api/theory` | Retrieves theoretical explanations using RAG over academic references (Gemini required). |
| `GET` | `/results/{session_id}` | Displays the final interview report. |

## Datasets

- **SQuAD (Stanford Question Answering Dataset)** — Reading comprehension dataset. Downloaded automatically from Hugging Face during indexing. `data/archive.zip` also stores a copy of the original v1.1 release.
- **CoachQuant** — Quantitative interview questions collected using a custom **Scrapy** crawler and normalized into JSONL format.

## Technology Stack

**Backend:** FastAPI · Pydantic · Jinja2 · Redis · Uvicorn

**RAG:** LangChain · ChromaDB · Google GenAI SDK (Gemini) · Groq · DeepSeek

**NLP:** Sentence-Transformers · KeyBERT · spaCy · SymPy · BERTScore
